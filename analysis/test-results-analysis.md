# Verifiable Credential フォーマット比較テスト結果 考察

## 1. 概要

本ドキュメントは、VC Comparison Tool（`/Users/naohirofujie/localrepos/RDF`）で実施した SD-JWT VC・JSON-LD VC・mdoc（ISO 18013-5）の3フォーマットに対するベンチマークおよびセキュリティテストの結果について考察する。テストは以下の3カテゴリで構成される。

1. **署名検証速度ベンチマーク** — 署名生成・検証のスループットとレイテンシ
2. **デシリアライズ複雑性分析** — コード量・非同期ステップ・循環的複雑度・パース時間
3. **正規化セキュリティテスト** — DoS・インジェクション・SSRF・アルゴリズム混同等の脆弱性評価

対象フォーマットの基本構成は以下の通りである。

| フォーマット | シリアライズ | 署名アルゴリズム | 正規化 | 主要規格 |
|---|---|---|---|---|
| SD-JWT VC | JWT コンパクト（JSON） | EdDSA (Ed25519) | なし | IETF RFC 9901 |
| JSON-LD VC | JSON テキスト | Ed25519 + SHA-256 | URDNA2015 (RDF) | W3C VCDM 2.0 |
| mdoc | CBOR バイナリ | ECDSA P-256 (ES256) | なし | ISO/IEC 18013-5 |

## 2. 署名検証速度ベンチマーク

### 2.1 テスト設計

各フォーマットについて、同一のクレデンシャル情報（氏名・生年月日・住所等）を対象に署名生成（sign）と署名検証（verify）を繰り返し実行し、平均レイテンシ（ms/op）とスループット（ops/sec）を計測した。ウォームアップを1回実行した後に本計測を行うことで、JIT最適化やキャッシュの初期化による偏りを低減している。

JSON-LD VC については、署名処理全体に加えて内部ステップ（正規化・ハッシュ・署名）の内訳も個別に計測した。

### 2.2 考察

**SD-JWT VC が最速となる構造的理由：** SD-JWT VC はJWTコンパクトシリアライゼーションを採用しており、署名対象データの前処理が不要である。`jose` ライブラリの `SignJWT` / `jwtVerify` の1回のAPI呼び出しで署名・検証が完結するため、オーバーヘッドが最小限に抑えられる。

**JSON-LD VC が最も遅い構造的理由：** JSON-LD VC の署名・検証フローには、URDNA2015による RDF正規化が必須のステップとして含まれる。正規化はブランクノードの同定（グラフ同型問題に帰着）を伴うアルゴリズムであり、単純なクレデンシャルであっても他の2フォーマットと比較して顕著なオーバーヘッドが発生する。ベンチマーク結果のbreakdownでは、正規化ステップが署名処理全体の大部分を占めることが確認できる。ハッシュ計算（SHA-256）やEd25519署名自体の所要時間は比較的小さい。

**mdoc は中間的なパフォーマンス：** mdoc は CBOR エンコード/デコード、COSE_Sign1 構造の構築、および各データ要素ごとのSHA-256ダイジェスト計算が必要である。CBOR はバイナリフォーマットであるため JSON パースより高速に動作するが、要素ごとのダイジェスト検証ループが追加的なコストとなる。ECDSA P-256 署名は WebCrypto API のネイティブ実装を使用しており、暗号処理自体は高速である。

**実用上の示唆：** モバイルデバイスや高頻度の検証が求められるユースケース（例：改札通過時のmDL提示）では、SD-JWT VC や mdoc のパフォーマンス優位が重要な選定要因となる。一方、JSON-LD VC はセマンティック相互運用性を重視する場面（分散型アイデンティティエコシステム間のクレデンシャル交換等）で選択される傾向があり、パフォーマンスはトレードオフとして許容される場合がある。

## 3. デシリアライズ複雑性分析

### 3.1 テスト設計

各フォーマットのデシリアライズ処理について、以下の静的・動的メトリクスを計測した。

| メトリクス | SD-JWT VC | JSON-LD VC | mdoc |
|---|---|---|---|
| コード行数 (LOC) | 10 | 35 | 25 |
| 非同期ステップ数 | 1 | 4 | 2 |
| 循環的複雑度 | 3 | 8 | 5 |
| 外部ネットワーク呼び出し | 0 | 2 | 0 |
| 外部依存ライブラリ | 1 (jose) | 4 (jsonld, DocumentLoader, sha256, ed25519) | 2 (cbor-x, WebCrypto) |

### 3.2 考察

**実装の単純性とセキュリティの関係：** SD-JWT VC のデシリアライズは10行程度で完結し、分岐は alg許可リスト・iss欠落・vct欠落の3箇所のみである。コード量が少ないことは、レビュー・監査のコストが低く、実装ミスに起因する脆弱性の混入リスクも低いことを意味する。

**JSON-LD VC の複雑性が突出している点：** JSON-LD VC は35行のコード、4つの非同期ステップ、循環的複雑度8と、3フォーマット中で最も複雑である。特に以下の2点が複雑性の主因である。

- **外部コンテキスト取得（2回のネットワーク呼び出し）：** `@context` に指定されたURLからコンテキスト定義をHTTPで取得する。本ツールでは `staticContexts.ts` に事前埋め込みした静的ローダーで代替しているが、本番環境ではネットワーク遅延・DNS障害・コンテキストサーバーのダウンが直接的な可用性リスクとなる。
- **URDNA2015正規化：** ブランクノードの正準ラベル付けアルゴリズムは、入力グラフの構造によっては指数的な計算量に達する。W3C RDFC-1.0仕様（§4.8.3）では呼び出し上限の設定を推奨しているが、多くの実装ではデフォルトで未設定である。

**mdoc のバランスの良さ：** mdoc は25行・非同期2ステップ・循環的複雑度5と中間的な複雑性を示す。CBOR デコードと COSE_Sign1 検証という明確に分離された2段階の処理であり、外部ネットワーク呼び出しが一切不要である点は運用上の大きな利点である。ただし、要素ごとのダイジェスト検証ループは要素数に比例するため、フィールド数が多いクレデンシャルでは相応の計算コストが発生する。

**開発者体験への影響：** SD-JWT VC は `jose` ライブラリの1APIで署名検証が完結するため、開発者の学習コストが低い。JSON-LD VC は JSON-LD プロセッサ、RDF正規化、暗号ライブラリの3層を理解する必要があり、参入障壁が高い。mdoc は CBOR/COSE という比較的ニッチな規格への知識が必要だが、処理フロー自体は直線的で理解しやすい。

## 4. 正規化セキュリティテスト

### 4.1 テスト項目と結果サマリ

| テスト項目 | 対象 | 深刻度 | 結果 |
|---|---|---|---|
| ポイズングラフ DoS (URDNA2015) | JSON-LD VC | HIGH〜CRITICAL | 脆弱（正常比 数倍〜数十倍の処理時間） |
| JSON-LD コンテキストインジェクション | JSON-LD VC | HIGH | 部分的（@protected 未設定で用語上書き可能） |
| リモートコンテキスト経由 SSRF | JSON-LD VC | CRITICAL | 脆弱（Document Loader許可リスト未実装時） |
| 正規化リスク非該当 | SD-JWT VC / mdoc | NONE | 該当なし |
| alg:none 攻撃 | SD-JWT VC | CRITICAL | 緩和済み（jose ライブラリが拒否） |
| アルゴリズム混同 RS256→EdDSA | SD-JWT VC | HIGH | 緩和済み（algorithms オプションで制限） |
| mdoc データ要素改ざん検出 | mdoc | HIGH | 緩和済み（SHA-256ダイジェスト検証で検出） |
| COSE プロテクトヘッダー改ざん | mdoc | HIGH | 緩和済み（Sig_Structure変更で署名不一致） |
| SSRF なし・ネットワーク取得なし | mdoc | NONE | 該当なし |

### 4.2 JSON-LD VC 固有のリスク考察

JSON-LD VC は3フォーマット中で唯一、構造的なセキュリティリスクを複数抱えている。これらはJSON-LD/RDFの設計思想そのものに起因するものであり、実装レベルの対策だけでは根本的には解消できない。

**ポイズングラフ DoS：** URDNA2015アルゴリズムはブランクノードの正準ラベル付けにおいて、最悪ケースで指数的な計算量を要する。テストでは20個のブランクノードで構成されたポイズングラフを入力した場合、通常のクレデンシャル正規化と比較して処理時間が大幅に増大することが確認された。実運用環境では、ブランクノード数やイテレーション回数に上限を設けるガードが必須である。

**コンテキストインジェクション：** JSON-LD の `@context` に攻撃者が用語定義を追加することで、`issuer` や `credentialSubject` といったフィールドの意味（IRI）を書き換えることが可能である。W3C VC Data Integrity仕様では `@protected` キーワードによる用語保護を推奨しているが、本テストではこの保護なしのコンテキストに対して正規化が成功し、用語の上書きが可能であることが確認された。

**SSRF（Server-Side Request Forgery）：** JSON-LD プロセッサは `@context` に指定されたURLに対してHTTPリクエストを送信する。攻撃者がクラウドメタデータエンドポイント（`http://169.254.169.254/latest/meta-data/`）や内部ネットワークのURLを `@context` に含めることで、SSRF攻撃が成立する。対策として、Document Loader に許可リスト（allowlist）を適用し、信頼できるコンテキストURLのみを許可する必要がある。本ツールでは `staticContexts.ts` にコンテキストを事前埋め込みすることで、ネットワーク取得自体を排除している。

### 4.3 SD-JWT VC のセキュリティ考察

SD-JWT VC に対するテストでは、`alg:none` 攻撃とアルゴリズム混同攻撃の両方が `jose` ライブラリによって適切に拒否された。これはRFC 8725（JWT Best Current Practices）に準拠した実装であることを示す。ただし、以下の点に注意が必要である。

- `jose` ライブラリ以外のJWT実装では `alg:none` を受理する既知の脆弱性（CVE-2015-9235）が存在する。ライブラリ選定時のセキュリティ実績が重要である。
- アルゴリズム混同攻撃の防御には、検証側で `algorithms` オプションを明示的に指定する必要がある。デフォルト設定に依存すると防御が不十分になる可能性がある。

### 4.4 mdoc のセキュリティ考察

mdoc は2つの改ざん検出テスト（データ要素改ざん、COSEヘッダー改ざん）の両方で改ざんを正しく検出し、「緩和済み」と判定された。

- **データ要素改ざん検出：** 各データ要素は個別にSHA-256ダイジェストが計算され、MSO（Mobile Security Object）内のダイジェスト値と照合される。要素値を改ざんするとダイジェストが一致しなくなるため、改ざんが検出される。
- **COSEヘッダー改ざん：** COSE_Sign1 の Sig_Structure にはプロテクトヘッダーが含まれるため、ヘッダーを改ざんすると署名検証が失敗する。RFC 9052 の設計により、アルゴリズムダウングレード攻撃が構造的に防止されている。

mdoc は外部URLの参照メカニズムを持たないため、JSON-LD 固有のSSRFやコンテキストインジェクションのリスクは構造的に存在しない。

## 5. 総合考察

### 5.1 フォーマット特性の三角トレードオフ

3フォーマットの比較から、以下のトレードオフ構造が浮かび上がる。

- **SD-JWT VC：** パフォーマンス最速・実装最シンプル・ライブラリエコシステムが成熟。ただしセマンティック相互運用性（RDFベースの意味的な型付け）を持たない。選択的開示はSD-JWT仕様（IETF RFC 9901）で実現可能。
- **JSON-LD VC：** W3C VCDMの正規実装であり、RDFに基づくセマンティック相互運用性を提供。代償として、パフォーマンス・実装複雑性・セキュリティの3軸すべてで劣位にある。特にURDNA2015正規化に起因するDoSリスクとSSRFリスクは構造的な問題であり、実装レベルの緩和策（呼び出し上限、Document Loader許可リスト等）が必須。
- **mdoc：** ISO/IEC 18013-5に準拠したモバイル身分証明書向けフォーマット。CBORバイナリの高効率なエンコードと、COSE_Sign1 + 要素別ダイジェストの堅牢なセキュリティモデルを兼ね備える。ネットワーク取得が不要な自己完結型設計。ただし、mDL（mobile Driving License）向けに最適化された設計であり、汎用VCとしての適用範囲はSD-JWT VCやJSON-LD VCより限定的。

### 5.2 ユースケース別の推奨

| ユースケース | 推奨フォーマット | 理由 |
|---|---|---|
| Webサービスでの認証・認可 | SD-JWT VC | パフォーマンス最優、JWTエコシステムの成熟度、既存OAuth/OIDC基盤との親和性 |
| 分散型IDエコシステム間連携 | JSON-LD VC | RDFベースのセマンティック相互運用性が必須の場面では代替困難（ただしセキュリティ対策を十分に実装すること） |
| モバイル身分証明書（mDL等） | mdoc | ISO 18013-5準拠・オフライン検証可能・バイナリ効率・政府機関での採用実績 |
| 高頻度・低レイテンシ検証 | SD-JWT VC / mdoc | JSON-LD VCの正規化オーバーヘッドは高頻度処理に不向き |
| セキュリティ最優先環境 | mdoc / SD-JWT VC | 外部ネットワーク依存がなく、攻撃面が構造的に小さい |

### 5.3 テスト手法の限界と今後の課題

本テストにはいくつかの制約がある。

- **ブラウザ環境での計測：** ベンチマークはブラウザ上の `performance.now()` で計測しており、GC（ガベージコレクション）やイベントループの影響を受ける。より正確な計測にはNode.js環境での `process.hrtime.bigint()` やサーバーサイドでの計測が望ましい。
- **ポイズングラフの規模：** 20ノードでの計測は概念実証としては十分だが、実際のDoS攻撃では数百〜数千のブランクノードが使用される可能性がある。より大規模な入力での挙動評価が必要。
- **静的コンテキストローダーの使用：** SSRF テストではリモートURLの存在を静的に分析しているが、実際のHTTPリクエスト送信は行っていない。実環境での検証にはサンドボックス内でのネットワーク挙動確認が必要。
- **選択的開示の未評価：** SD-JWT VCの選択的開示（Selective Disclosure）やmdocの要素単位開示のパフォーマンスは本テストの対象外である。実用上は開示プロトコルのオーバーヘッドも考慮すべきである。
- **テスト結果の永続化：** 現時点ではベンチマーク結果はブラウザの React state に保持されるのみで、ディスクへのエクスポート機能が未実装である。再現性のある比較のためには、結果のJSON/CSV出力機能の追加が望ましい。

## 6. 結論

SD-JWT VC・JSON-LD VC・mdoc の3フォーマットは、それぞれ異なる設計哲学とトレードオフを持つ。パフォーマンスと実装の単純性では SD-JWT VC が優位であり、セマンティック相互運用性では JSON-LD VC が唯一の選択肢であり、モバイル身分証明書の領域では mdoc がISO標準として確立されている。

セキュリティの観点では、JSON-LD VC はRDF正規化に起因する構造的リスク（ポイズングラフDoS、コンテキストインジェクション、SSRF）を抱えており、運用には十分な緩和策の実装が前提となる。SD-JWT VC と mdoc は適切なライブラリ選定とパラメータ設定により、テストで検証した攻撃ベクトルに対して十分な耐性を示した。

フォーマット選定においては、対象ユースケースの要件（相互運用性・パフォーマンス・セキュリティ・規制準拠）を総合的に評価し、トレードオフを明示的に受容した上で判断することが重要である。
